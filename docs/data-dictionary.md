# Athlete Portal — Data Dictionary

> **The live Drizzle schema is the source of truth.** This doc captures the *conventions* every schema file is expected to follow. The per-table column detail lives in the schema files themselves under [`packages/shared/src/db/schema/`](../packages/shared/src/db/schema/) — chasing column lists here means they go stale.
>
> Each domain module exports the tables, relations, and `$inferSelect` / `$inferInsert` types consumed elsewhere via `@repo/shared/db`. The module list accrues as Epics deliver — there is no canonical list to maintain in this file.

## Naming conventions

- **Table names** are `snake_case`, **plural** (`users`, `organizations`, `team_invites`).
- **Column names** are `snake_case`. The TypeScript identifier in Drizzle is `camelCase` (`createdAt` → `created_at`).
- **Foreign-key columns** are `<singular>_id` (`organization_id`, `user_id`).
- **Boolean columns** read as predicates (`is_public`, `is_archived`), never negative predicates (`is_not_deleted` ❌).
- **Domain modules** under `packages/shared/src/db/schema/` are named for the bounded context they own, not for their plural (`identity.ts`, `teams.ts`, `events.ts`, `social.ts`).

## Primary keys

- `text('id')` storing a UUIDv4 generated by the application unless the table notes otherwise.
- Generated client-side via the shared `newId()` helper so tests and seeds are deterministic when a fixed RNG is injected.

## Timestamps (audit fields)

Every persisted row carries:

```ts
created_at: text('created_at')
  .notNull()
  .default(sql`(strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))`),
updated_at: text('updated_at')
  .notNull()
  .default(sql`(strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))`),
```

- ISO-8601 UTC strings. **Never** local time, never Unix epoch.
- `updated_at` is set by the application on every write — Drizzle `$onUpdate` is *not* a substitute (libSQL has no row-level triggers).
- Rows that participate in soft-delete also carry `deleted_at` (nullable) — see below.

## Booleans

- Stored as `integer(name, { mode: 'boolean' })` per SQLite/libSQL convention. Drizzle handles the encode/decode.
- Default explicitly to `false` (`.default(false)`) — never `null` for a boolean unless the column genuinely represents *unknown*.

## Foreign keys

- Prefer `onDelete: 'cascade'` for **owned** children (e.g. `team_invites.team_id → teams.id` cascades).
- Use `onDelete: 'set null'` only when the relationship is **informational** (e.g. `events.venue_id → venues.id` may set null if the venue is deleted but the event endures).
- Never use `onDelete: 'restrict'` — it produces opaque failure modes at the API boundary. Validate at the query layer instead.

## Soft delete + tombstoning

User-visible content (`highlights`, `comments`, posts, profile assets, etc.) uses **soft delete**:

```ts
deleted_at: text('deleted_at'),  // nullable, ISO-8601 when set
```

- All read queries filter `deleted_at IS NULL`. There are no exceptions — analytics that need to count tombstones do so via a separate query that explicitly opts in.
- Cascade-deletion of child rows on parent delete is a **write-time responsibility of the query module** (see ADR-001), not a DB trigger. The query module stamps `deleted_at` on every affected row in a transaction.
- For privacy-sensitive surfaces (account closure, DSAR right-to-erasure), soft delete is paired with **cryptographic tombstoning**: PII columns are overwritten with `NULL` and a `tombstone_reason` column records the lawful basis. Recoverability is intentionally lost.

## Privacy-gated columns

- Visibility fields are typed `'public' | 'unlisted' | 'private'` (string union enforced by the Zod schema; stored as `text` with a `CHECK` constraint).
- Field-level visibility (e.g. "show my last name to followers only") lives in a `privacy_settings` JSON column on `users`, not as per-field columns.
- Any column containing personal data is enumerated in the redaction allowlist at [`packages/shared/src/observability/redaction.ts`](../packages/shared/src/observability/redaction.ts) — the **edge redactor** is the contract, not Sentry's server-side scrubbing.

## RBAC scoping columns

- Resources owned by an organization carry `organization_id` (FK, indexed). Resources owned by a team carry both `organization_id` *and* `team_id` (denormalized for query-side authorization filtering).
- Resources owned by a user carry `user_id`. Composite ownership (e.g. a post by a user within a team) carries all three.
- The RBAC policy module at `packages/shared/src/rbac/` reads these columns; routes never re-derive ownership from joins.

## Identity axes — privilege role vs team-graph membership

Identity is expressed across **two orthogonal axes** (see [ADR-022](./decisions.md#adr-022--privilege-role-and-team-graph-membership-are-orthogonal-role-escalation-is-invitation-only)). They are different columns in different tables — do not conflate them.

- **Privilege** — `users.role` (`text NOT NULL DEFAULT 'member'`): `dev_admin | org_admin | team_admin | member`. Answers "what administrative capability does this user have?" `member` is the no-admin baseline.
- **Team-graph position** — carried by join rows, **not** by `users.role`:
  - `coach_assignments` — `(org_id, team_id, coach_user_id, ended_at)`. A user **is a coach** of a team iff an active (`ended_at IS NULL`) row exists; authorization is derived via `requireCoachOnTeam`, never from `users.role`. A coach's privilege role is `member`.
  - `athlete_memberships` — `(org_id, team_id, athlete_user_id, ended_at)`. A user **is an athlete** on a team iff an active row exists.
- **`member` ≠ "athlete".** Parents (`parent_athlete_links`) and signed-in users not on any roster are also `member`. "Athlete" is a **derived persona label** surfaced in UI copy, not a stored role — store `member`, display "Athlete".
- **Role escalation is invitation-only.** Self-signup users default to `member`; `/onboarding` never writes `role`. Coach/athlete memberships are created at invitation-accept; `org_admin` / `dev_admin` are bootstrapped out-of-band. `invitations.role` is a frozen `['coach','athlete']` tuple and cannot mint an admin.
- **Extension point (not yet built):** coach designations (head / assistant / sport-specific) will be an **additive, nullable** `coach_assignments.designation` column (`null` = unspecified coach). The per-team join already supports it without restructuring; deferred to the coach-management Epic per ADR-022.

## Validation contract

- Every column that crosses a system boundary has a Zod schema in [`packages/shared/src/schemas/`](../packages/shared/src/schemas/).
- The Zod schema is **authoritative for boundary validation**. The Drizzle definition is **authoritative for storage shape**. They are kept in lockstep — never validate at the boundary against `inferSelect`.
- Schemas re-export `*Input` (for `POST`/`PATCH`) and `*Output` (for response shaping) variants. `Output` always strips internal fields (`deleted_at`, raw hashes, internal IDs not intended for clients).

## Indexes

- Every FK column is indexed. The `(organization_id, deleted_at)` and `(team_id, deleted_at)` composite indexes are common — soft-delete filtering and tenant scoping always travel together.
- Unique constraints on `(slug)` columns (per ADR-008) carry a partial index excluding tombstoned rows so slug reuse is possible after deletion.

## Seeds

- Deterministic seed data lives at [`packages/shared/src/db/seed.ts`](../packages/shared/src/db/seed.ts).
- **Never modify seed data without explicit approval** — it is referenced by acceptance scenarios, fixture builders, and the dev environment. Drift here causes silent test failures across the pyramid.
- Test fixtures (one-row builders for contract/unit tests) live separately at `packages/shared/src/testing/factories/` and follow ADR-004.

## Migrations

- Generated via Drizzle's `drizzle-kit generate` workflow. Never hand-edit a migration file after it's committed.
- Migration files are append-only — a column rename is a **new migration** that adds the new column, backfills, then drops the old one in a follow-up migration once consumers have caught up.
- Destructive migrations (drop column, drop table, rename with data loss) are flagged by the planning risk-heuristics and require explicit `agent::blocked` resume before running in any non-local environment.

## Roster tables (Epic #11)

The two persisted entities introduced by Epic #11's coach-roster surface. Both follow the cross-tenant pattern established by migration `0002` for `coach_assignments` / `athlete_memberships` (denormalised `org_id`, indexed FKs, ABORT triggers on org-id mismatch). See migration [`packages/shared/src/db/migrations/0007_roster.sql`](../packages/shared/src/db/migrations/0007_roster.sql) for the authoritative DDL.

### `roster_entry`

One row per athlete-on-this-team. Created when a roster invite is accepted; soft-deleted via `ended_at` when the coach removes the athlete.

| Column | Type | Notes |
| --- | --- | --- |
| `id` | text PK | |
| `org_id` | text NOT NULL | Denormalised for `scopedDb(actor)`; FK → `organizations.id`. |
| `team_id` | text NOT NULL | FK → `teams.id`. |
| `athlete_user_id` | text NOT NULL | FK → `users.id`. |
| `jersey_number` | text NULL | CHECK matches `^[0-9]{1,3}$` (keeps `"00"` and leading zeros representable); soft uniqueness — same number can be reused across teams. |
| `primary_position` | text NULL | CHECK `length <= 32`. |
| `ended_at` | integer NULL | Soft-delete tombstone (unixepoch). |
| `created_at`, `updated_at` | integer NOT NULL | Audit fields per § Timestamps. |

Indexes: `(org_id, team_id)`, `(team_id, ended_at)`, and a **partial unique** `(team_id, athlete_user_id) WHERE ended_at IS NULL` — an athlete is on a team at most once while the membership is active.

### `roster_invite`

One row per outstanding coach-issued invitation. Distinct from `invitations` (Clerk-mediated org-admin invites) per [ADR-021](./decisions.md#adr-021--roster-invites-are-separate-from-clerk-org-admin-invitations).

| Column | Type | Notes |
| --- | --- | --- |
| `id` | text PK | |
| `org_id` | text NOT NULL | Denormalised; FK → `organizations.id`. |
| `team_id` | text NOT NULL | FK → `teams.id`. |
| `email` | text NOT NULL | Lowercased application-side; CHECK refuses obvious junk and caps length at 254. |
| `first_name`, `last_name` | text NULL | Optional pre-fill for the public accept page. |
| `token_hash` | text NOT NULL | The only authorization on the public accept route; UNIQUE indexed for constant-time lookup. The raw token is never persisted. |
| `status` | text NOT NULL | One of `pending` / `accepted` / `declined` / `expired` / `revoked`; default `'pending'`. Expiry is **lazy** — a row stays `pending` until a read or accept attempt finds `expires_at < now()` and updates `status` to `'expired'` in the same transaction. No nightly cron. |
| `expires_at` | integer NOT NULL | Unixepoch; the 7-day TTL is set application-side at invite creation. |
| `accepted_at`, `declined_at` | integer NULL | Set on the corresponding lifecycle transition. |
| `invited_by_user_id` | text NOT NULL | FK → `users.id`. |
| `created_at`, `updated_at` | integer NOT NULL | Audit fields per § Timestamps. |

Indexes: `(org_id)`, `(team_id, status)`, `(email)`, and the unique `(token_hash)`.
